Manual de Google BigQuery

¿Qué es BigQuery?

Google BigQuery es un almacén de datos (data warehouse) serverless y escalable de Google Cloud Platform. Es perfecto para análisis de grandes volúmenes de datos mediante SQL estándar.

Características principales

  • Serverless: no necesitas gestionar servidores
  • Escalable: soporta desde GB hasta petabytes de datos
  • SQL estándar: usa sintaxis SQL ANSI compatible con la mayoría de bases de datos
  • Rápido: queries distribuidas que procesan datos en paralelo

Conceptos básicos

Estructura jerárquica

Proyecto GCP
 └── Dataset
      └── Tabla
           └── Campos (columnas)
  • Proyecto: tu contenedor de GCP (ej: mi-proyecto-bootcamp)
  • Dataset: agrupación lógica de tablas (ej: Tu_empresa)
  • Tabla: estructura de datos con filas y columnas (ej: customers)

Tipos de dato en BigQuery

Tipo BigQuery Equivalente Python Ejemplo
STRING str 'Madrid', 'usuario@email.com'
INT64 int 42, 1000, -5
FLOAT64 float 99.99, -3.14, 1.5
BOOL bool TRUE, FALSE
DATE datetime.date '2024-12-31'
TIMESTAMP datetime.datetime '2024-12-31 14:30:00 UTC'
En BigQuery NO existen VARCHAR ni INT. Usa siempre STRING e INT64.

  1. Ir a IAM & Admin → Service Accounts
  2. Clic en Create Service Account
  3. Nombre: bigquery-admin
  4. Asignar rol: BigQuery Admin
  5. Crear una clave JSON y descargarla

2. Configurar variables de entorno

Crear un fichero .env en la raíz del proyecto:

El fichero .env NUNCA se sube a Git. Debe estar en .gitignore.

Crear dataset y tablas con Python

Instalación de librerías

pip install google-cloud-bigquery google-auth python-dotenv

Código para crear el dataset

from google.cloud import bigquery
from google.oauth2 import service_account
import os
from dotenv import load_dotenv

load_dotenv()

# Configuración
PROJECT_ID = os.getenv("GCP_PROJECT_ID")
DATASET_ID = os.getenv("BQ_DATASET_ID")
CREDENTIALS_PATH = os.getenv("GOOGLE_APPLICATION_CREDENTIALS")

# Cliente autenticado
credentials = service_account.Credentials.from_service_account_file(
    CREDENTIALS_PATH
)
client = bigquery.Client(
    project=PROJECT_ID,
    credentials=credentials
)

# Crear dataset
dataset_ref = bigquery.Dataset(f"{PROJECT_ID}.{DATASET_ID}")
dataset_ref.location = "EU"  # Datos en Europa
dataset = client.create_dataset(dataset_ref, exists_ok=True)
print(f"Dataset {DATASET_ID} creado")

Código para crear una tabla

schema = [
    bigquery.SchemaField("customer_id", "INT64", mode="REQUIRED"),
    bigquery.SchemaField("first_name", "STRING", mode="REQUIRED"),
    bigquery.SchemaField("last_name", "STRING", mode="REQUIRED"),
    bigquery.SchemaField("email", "STRING", mode="REQUIRED"),
    bigquery.SchemaField("country", "STRING", mode="NULLABLE"),
    bigquery.SchemaField("registration_date", "DATE", mode="REQUIRED"),
]

table_ref = f"{PROJECT_ID}.{DATASET_ID}.customers"
table = bigquery.Table(table_ref, schema=schema)
table = client.create_table(table, exists_ok=True)
print(f"Tabla customers creada")

Cargar datos desde pandas

import pandas as pd

# Crear DataFrame de ejemplo
df = pd.DataFrame({
    'customer_id': [1, 2, 3],
    'first_name': ['Ana', 'Carlos', 'María'],
    'last_name': ['García', 'López', 'Fernández'],
    'email': ['ana@email.com', 'carlos@email.com', 'maria@email.com'],
    'country': ['Spain', 'Spain', 'Mexico'],
    'registration_date': pd.to_datetime(['2024-01-15', '2024-02-20', '2024-03-10']).date
})

# Configurar la carga
job_config = bigquery.LoadJobConfig(
    write_disposition="WRITE_TRUNCATE",  # Sobreescribe
    schema=schema
)

# Cargar datos
table_ref = f"{PROJECT_ID}.{DATASET_ID}.customers"
job = client.load_table_from_dataframe(df, table_ref, job_config=job_config)
job.result()  # Esperar a que termine
print(f"{len(df)} filas cargadas")

Ejecutar queries

Query básica

query = f"""
    SELECT 
        country,
        COUNT(*) as total_customers
    FROM `{PROJECT_ID}.{DATASET_ID}.customers`
    GROUP BY country
    ORDER BY total_customers DESC
"""

# Ejecutar y obtener resultados
df_result = client.query(query).to_dataframe()
print(df_result)

Query con JOINs

query = f"""
    SELECT 
        c.first_name,
        c.last_name,
        COUNT(o.order_id) as total_orders,
        SUM(p.amount) as total_spent
    FROM `{PROJECT_ID}.{DATASET_ID}.customers` c
    LEFT JOIN `{PROJECT_ID}.{DATASET_ID}.orders` o 
        ON c.customer_id = o.customer_id
    LEFT JOIN `{PROJECT_ID}.{DATASET_ID}.payments` p 
        ON o.order_id = p.order_id
    WHERE p.status = 'completed'
    GROUP BY c.first_name, c.last_name
    ORDER BY total_spent DESC
    LIMIT 10
"""

top_customers = client.query(query).to_dataframe()
print(top_customers)
BigQuery usa backticks ` en lugar de comillas para los nombres de tablas con puntos: `proyecto.dataset.tabla`

Manual de Entornos Virtuales (venv)

¿Qué es un entorno virtual?

Un entorno virtual es un directorio aislado que contiene una instalación independiente de Python y sus librerías. Te permite tener diferentes versiones de paquetes para diferentes proyectos sin conflictos.

¿Por qué usar entornos virtuales?

  • Aislamiento: cada proyecto tiene sus propias dependencias
  • Reproducibilidad: cualquiera puede recrear el entorno exacto
  • Sin conflictos: evitas problemas de versiones incompatibles
  • Limpieza: no "contaminas" la instalación global de Python
Nunca trabajes directamente con la instalación global de Python para proyectos. Siempre usa entornos virtuales.

Crear un entorno virtual

Desde la terminal, en la raíz del proyecto:

python -m venv venv

Esto crea una carpeta venv/ con todo lo necesario.

El primer "venv" es el módulo de Python. El segundo es el nombre de la carpeta (puedes usar otro nombre, pero "venv" es convención).

Activar el entorno virtual

En Mac/Linux:

source venv/bin/activate

En Windows (cmd):

venv\Scripts\activate

En Windows (PowerShell):

venv\Scripts\Activate.ps1

Cuando esté activado, verás (venv) al principio de la línea de comandos:

(venv) usuario@pc:~/proyecto$

Instalar paquetes

Instalar un paquete individual:

pip install pandas

Instalar todas las dependencias del proyecto:

pip install -r requirements.txt

Ver paquetes instalados:

pip list

Gestionar requirements.txt

El fichero requirements.txt lista todas las dependencias del proyecto con sus versiones exactas.

Generar requirements.txt:

pip freeze > requirements.txt

Ejemplo de requirements.txt:

google-cloud-bigquery==3.17.2
pandas==2.2.1
faker==24.2.0
python-dotenv==1.0.1
Ejecuta pip freeze > requirements.txt cada vez que instales un paquete nuevo. Así tus compañeros podrán sincronizar las dependencias.

Desactivar el entorno virtual

deactivate

Esto te devuelve al entorno global de Python.

Workflow completo con venv

Setup inicial (primera vez)

# 1. Crear el entorno virtual
python -m venv venv

# 2. Activarlo
source venv/bin/activate  # Mac/Linux
venv\Scripts\activate     # Windows

# 3. Instalar dependencias
pip install -r requirements.txt

# 4. Verificar
pip list

Trabajo diario

# Al empezar a trabajar
source venv/bin/activate

# ... trabajar en el proyecto ...

# Si instalas un paquete nuevo
pip install nueva-libreria
pip freeze > requirements.txt

# Al terminar (opcional)
deactivate

Sincronizar con el equipo

# Cuando un compañero añade dependencias nuevas
git pull origin main
pip install -r requirements.txt

.gitignore y entornos virtuales

La carpeta venv/ NUNCA se sube a Git porque:

  • Pesa mucho (cientos de MB)
  • Es específica de cada máquina
  • Se puede recrear fácilmente desde requirements.txt

Asegúrate de que tu .gitignore contiene:

venv/
.venv/
env/
.env
Si accidentalmente subiste venv/ a Git, puedes eliminarlo del historial con: 
git rm -r --cached venv/
git commit -m "remove venv from tracking"

Manual de Git Flow Simplificado

¿Qué es Git Flow?

Git Flow es una metodología de trabajo con Git que define cómo organizar las ramas y coordinar el trabajo en equipo.

Para este proyecto usamos una versión simplificada: solo trabajamos con main y ramas de feature. No usamos ramas develop, release ni hotfix.

Estructura de ramas

main

Rama principal y protegida. Contiene solo código revisado y estable.

  • Nunca se hace commit directo a main
  • Solo se integran cambios vía Pull Request
  • Requiere aprobación de otro integrante

feature/nombre-funcionalidad

Ramas temporales para desarrollar funcionalidades.

  • Se crean desde main
  • Cada integrante trabaja en su propia rama
  • Se eliminan tras hacer merge

Setup inicial del repositorio

1. Una persona crea el repositorio en GitHub

  1. New repository → nombre: tc-sql-Tu_empresa
  2. Inicializar con README
  3. Settings → Collaborators → Add people (añadir al equipo)

2. Proteger la rama main

  1. Settings → Branches → Add rule
  2. Branch name pattern: main
  3. Marcar:
    • ✅ Require a pull request before merging
    • ✅ Require approvals: 1
    • ✅ Dismiss stale pull request approvals when new commits are pushed
  4. Save changes
Proteger main desde el principio evita commits accidentales y fuerza el flujo de PR.

3. Todos clonan el repositorio

git clone https://github.com/usuario/tc-sql-Tu_empresa.git cd tc-sql-Tu_empresa

Workflow de desarrollo

PASO 1: Crear una rama de feature

# Asegurarse de estar en main y actualizado
git checkout main
git pull origin main

# Crear nueva rama de feature
git checkout -b feature/bigquery-setup-ana

# Verificar en qué rama estás
git branch
Convención de nombres: feature/descripcion-nombre
Ejemplos:
  • feature/er-diagram-carlos
  • feature/data-generation-maria
  • feature/queries-validation-pablo

PASO 2: Hacer commits

# Ver cambios
git status

# Añadir ficheros
git add notebooks/01_setup_bigquery.ipynb
git add requirements.txt

# O añadir todos los cambios
git add .

# Hacer commit con mensaje descriptivo
git commit -m "feat: crear dataset y tabla customers en BigQuery"

Formato de commits: Conventional Commits

Tipo Cuándo usarlo Ejemplo
feat: Nueva funcionalidad feat: añadir generación de datos de products
fix: Corrección de bug fix: corregir FK de order_items a orders
docs: Documentación docs: añadir justificación de 3NF
refactor: Refactorización refactor: extraer generación de fechas a función
style: Formato código style: formatear queries con black

PASO 3: Subir la rama al repositorio

git push origin feature/bigquery-setup-ana

PASO 4: Abrir Pull Request en GitHub

  1. Ir al repositorio en GitHub
  2. Aparecerá un banner "Compare & pull request" → clic
  3. Rellenar el PR:
    • Título: descripción clara de los cambios
    • Descripción: qué se ha hecho, por qué, cómo probarlo
    • Reviewers: asignar a un compañero del equipo
  4. Create pull request

Plantilla de descripción de PR

## Qué hace este PR
- Crea el dataset Tu_empresa en BigQuery
- Define el esquema de la tabla customers
- Implementa función de carga de datos

## Cómo probarlo
1. Activar entorno virtual
2. Configurar credenciales en .env
3. Ejecutar notebook 01_setup_bigquery.ipynb

## Checklist
- [x] Código testeado localmente
- [x] requirements.txt actualizado
- [x] .env.example actualizado si añadí variables

PASO 5: Code Review

El compañero asignado revisa el código y:

  • Lee los cambios en la pestaña "Files changed"
  • Deja comentarios si hay dudas o mejoras
  • Aprueba con "Approve" si todo está correcto
  • Puede pedir cambios con "Request changes"
Cosas a revisar en un PR:
  • ¿El código funciona?
  • ¿Está bien comentado?
  • ¿Sigue las convenciones del proyecto?
  • ¿Los nombres de variables son claros?
  • ¿Falta algo en requirements.txt?

PASO 6: Merge del PR

Una vez aprobado:

  1. En GitHub, en el PR, clic en "Squash and merge"
  2. Confirmar el merge
  3. Borrar la rama de feature (GitHub lo ofrece automáticamente)
Usamos Squash and merge para condensar todos los commits de la rama en uno solo. Esto mantiene el historial de main limpio y fácil de leer.

PASO 7: Sincronizar main localmente

# Volver a main
git checkout main

# Traer los últimos cambios
git pull origin main

# Borrar la rama local de feature (ya está mergeada)
git branch -d feature/bigquery-setup-ana

Manejo de conflictos

Un conflicto ocurre cuando dos personas modifican las mismas líneas de código.

Si hay conflicto al hacer merge:

# 1. Actualizar tu rama con los últimos cambios de main
git checkout feature/tu-rama
git pull origin main

# 2. Git te avisará de los conflictos
# Abrir los ficheros marcados y resolver manualmente

# 3. Los conflictos se ven así:
<<<<<<< HEAD
código de tu rama
=======
código de main
>>>>>>> main

# 4. Elegir qué código mantener y borrar las marcas

# 5. Añadir los ficheros resueltos
git add fichero_resuelto.py

# 6. Hacer commit
git commit -m "fix: resolver conflictos con main"

# 7. Subir
git push origin feature/tu-rama
Para evitar conflictos, sincroniza tu rama con main frecuentemente: 
git checkout feature/tu-rama
git pull origin main

Comandos Git de referencia rápida

Comando Descripción
git status Ver estado actual y cambios pendientes
git branch Listar ramas locales
git checkout main Cambiar a la rama main
git checkout -b feature/X Crear y cambiar a nueva rama
git add . Añadir todos los cambios
git commit -m "mensaje" Guardar cambios con mensaje
git push origin rama Subir rama al repositorio remoto

Gestión de Costes en BigQuery

Este proyecto está diseñado para NO generar ningún coste. BigQuery tiene un Free Tier permanente más que suficiente para este Team Challenge.

Free Tier permanente de BigQuery

Google Cloud ofrece un Free Tier permanente (no es trial temporal) que incluye cada mes:

Recurso Límite gratuito mensual Tu proyecto (estimado)
Almacenamiento 10 GB ~5-15 MB (500× menos)
Procesamiento de queries 1 TB (1000 GB) ~100 MB (10000× menos)
Streaming inserts 10 GB No aplicable (usáis batch)

¿Cuánto ocupan vuestros datos?

Con los volúmenes recomendados del TC (500 clientes, 70 productos, 2000 pedidos, etc.):

  • Almacenamiento total: aproximadamente 10-15 MB
  • Queries de verificación: procesarán unos 50-100 MB de datos
  • Coste real: 0€ — estáis MUY por debajo del límite gratuito

Buenas prácticas para evitar costes inesperados

1. Configurar alertas de presupuesto

Configurad una alerta en $0 para recibir un aviso si os acercáis al límite del Free Tier:

  1. Ir a Billing en la consola de GCP
  2. Clic en Budgets & alerts
  3. Create Budget
  4. Establecer límite: $0
  5. Alertas al 50%, 90% y 100%

2. Buenas prácticas con queries

NUNCA hacer:
  • SELECT * FROM tabla sin LIMIT en tablas grandes
  • Queries exploratorias sin verificar el estimado de datos procesados
  • Scheduled queries que ejecuten cada pocos minutos
SIEMPRE hacer:
  • Usar LIMIT 100 cuando exploréis datos
  • Usar el botón Preview en la consola (gratis, no consume cuota)
  • Revisar el estimado "This query will process X MB" antes de ejecutar
  • Usar filtros WHERE para reducir datos procesados

3. Monitorizar el consumo

Para ver cuánto habéis consumado:

  1. Ir a Billing → Reports
  2. Filtrar por servicio: BigQuery
  3. Ver gráfico de consumo mensual

Al finalizar el bootcamp

Opción 1: Mantener el dataset (recomendada)

NO hace falta borrar nada. Vuestros datos ocupan tan poco (<15 MB) que podéis dejarlo indefinidamente sin coste. BigQuery NO cobra por tablas inactivas dentro del Free Tier.

Podéis mantener vuestro proyecto como portfolio para mostrar en entrevistas de trabajo.

Opción 2: Eliminar el dataset

Si preferís limpiar todo al terminar:

bq rm -r -f proyecto:Tu_empresa

O desde la consola:

  1. BigQuery → vuestro dataset
  2. Clic en los 3 puntos → Delete dataset
  3. Confirmar

Lo que NO tenéis que hacer

A diferencia de servicios como EC2 o Cloud SQL:

  • ❌ NO hay que "pausar" BigQuery — no existe ese concepto
  • ❌ NO hay que "apagar instancias" — BigQuery es serverless
  • ❌ NO hay cargos por tener tablas creadas — solo pagas almacenamiento (gratis hasta 10 GB)
  • ❌ NO hay cargos por hora — solo pagas por queries ejecutadas

Servicios que SÍ tienen coste (evitadlos)

NO activéis estos servicios a menos que sepáis lo que hacéis:
  • BigQuery BI Engine: tiene coste mensual
  • BigQuery ML: tiene costes adicionales
  • Compute Engine / Cloud SQL: son servicios diferentes con cargo
  • Cloud Storage (más de 5 GB): tiene Free Tier menor

Resumen

Acción ¿Tiene coste?
Crear dataset y tablas ❌ No (gratis)
Almacenar 10-15 MB de datos ❌ No (gratis hasta 10 GB)
Ejecutar queries del TC ❌ No (gratis hasta 1 TB procesado)
Tener las tablas creadas sin usar ❌ No (gratis)
Subir un archivo de 50 GB ⚠️ Sí (supera Free Tier)
Ejecutar queries que procesen 2 TB ⚠️ Sí (supera Free Tier)

Solución rápida a errores comunes

  • ModuleNotFoundError: instalar librería con pip install nombre
  • Permission denied en Git: verificar permisos de colaborador en GitHub
  • Credenciales BigQuery inválidas: revisar ruta en GOOGLE_APPLICATION_CREDENTIALS
  • Conflicto al hacer merge: hacer git pull origin main en tu rama y resolver manualmente
  • venv no se activa: verificar que estás en la carpeta correcta
  • "This query will process X GB": añadir WHERE o LIMIT para reducir datos procesados
