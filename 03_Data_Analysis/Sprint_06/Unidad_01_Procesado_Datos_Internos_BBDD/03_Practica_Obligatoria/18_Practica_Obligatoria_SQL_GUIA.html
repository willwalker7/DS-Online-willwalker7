Ejercicio 0 Imports, conexión y función helper ⏱ 5 min
Importa los paquetes necesarios, conecta a chinook.db y crea una función que facilite la ejecución de queries.
Explicación: Este ejercicio es puro boilerplate pero sienta las bases del notebook. Hay dos formas válidas de crear la función helper: usando el cursor directamente (más manual) o usando pd.read_sql (más limpio y directo). Ambas son correctas; en el curso se usan las dos para ver la diferencia. El flag copy_on_write=True prepara Pandas para el comportamiento por defecto de la versión 3.0.
¿Qué es un cursor?
Cuando Python se conecta a una base de datos, la conexión (conn) es el canal de comunicación — como abrir la puerta a la BD. El cursor es el puntero que se mueve por ese canal para enviar comandos y leer resultados fila a fila.

Puedes imaginarlo como un cursor de texto en un editor: la conexión es el documento abierto, y el cursor es donde ocurre la acción. Cada vez que llamas a cursor.execute(query) mandas una instrucción SQL; con cursor.fetchall() recoges todas las filas devueltas; con cursor.description obtienes los nombres de las columnas.

Una misma conexión puede tener varios cursores abiertos a la vez, lo que permite ejecutar queries independientes en paralelo sin interferencia.
¿Por qué sqlite3 y no otro motor?
sqlite3 viene incluido en la librería estándar de Python — no hay que instalar nada. SQLite guarda toda la base de datos en un único fichero .db, lo que lo hace ideal para prácticas, prototipos y proyectos pequeños: sin servidor, sin configuración, sin permisos de red.

En producción se usan motores como PostgreSQL o MySQL, que soportan múltiples usuarios concurrentes, mayor volumen de datos y más funcionalidades avanzadas. La sintaxis SQL que aprenderéis aquí es prácticamente idéntica en todos ellos; la única diferencia notable con SQLite es que algunas funciones de cadena o fecha cambian de nombre. 
import pandas as pd
import sqlite3

pd.options.mode.copy_on_write = True  # CoW por defecto en Pandas 3.0+

# Conexión a la base de datos
conn = sqlite3.connect("./data/chinook.db")
cursor_ejer = conn.cursor()

# Opción 1 — con cursor
def sql_query(query):
    cursor_ejer.execute(query)
    resultado = cursor_ejer.fetchall()
    columnas = [d[0] for d in cursor_ejer.description]
    return pd.DataFrame(resultado, columns=columnas)

# Opción 2 — con pd.read_sql (más recomendable)
def sql_query(query):
    return pd.read_sql(query, conn)
Punto clave en corrección: Que exista la función helper (de cualquiera de las dos formas) y que la conexión use la ruta correcta.

1.3 Clientes de USA por estado (COUNT + GROUP BY) ⏱ 5 min
Muestra cuántos clientes de Estados Unidos hay por estado, ordenado de mayor a menor.
Conceptos clave:
  • GROUP BY + COUNT: agrupa las filas por un campo y cuenta cuántas hay en cada grupo. COUNT(1) es equivalente a COUNT(*) — ver explicación detallada abajo.
  • ORDER BY: en SQLite podemos usar la posición numérica de la columna (ORDER BY 2 DESC) o su alias si le hemos dado uno.
  • Pandas: .groupby() + .count() + .sort_values(ascending=False).
💡 ¿Por qué COUNT(1) o COUNT(*) y no COUNT(columna)?

En SQL hay tres formas habituales de contar, y no son idénticas:

Forma Qué cuenta ¿Ignora NULLs?
COUNT(*) Todas las filas del grupo, sin excepción No — cuenta incluso las filas con NULLs en cualquier columna
COUNT(1) Evalúa el literal 1 por cada fila — siempre distinto de NULL No — idéntico en la práctica a COUNT(*)
COUNT(columna) Solo los valores no nulos de esa columna Sí — si la columna tiene NULLs, el resultado será menor

En este ejercicio queremos contar cuántos clientes hay por estado, es decir, simplemente cuántas filas pertenecen a cada grupo — independientemente de si algún campo tiene NULL. Por eso COUNT(*) y COUNT(1) son la opción correcta.
Si usásemos COUNT(State) obtendríamos el mismo resultado aquí porque State no tiene NULLs en los clientes de USA, pero si los tuviese el conteo sería incorrecto. La convención más extendida hoy es usar COUNT(*); COUNT(1) era una optimización en bases de datos antiguas que los motores modernos (incluido SQLite) ya no necesitan.

Inciso Joins en Pandas — el método merge() ⏱ 5 min
Resumen del método merge():
  • how="left" → LEFT JOIN
  • how="right" → RIGHT JOIN
  • how="inner" → INNER JOIN (por defecto)
  • how="outer" → FULL OUTER JOIN
  • on="columna" cuando la columna de cruce se llama igual en ambos DataFrames
  • left_on / right_on cuando los nombres son distintos
  • left_index / right_index = True para cruzar por el índice
  • suffixes=("_x","_y") controla los sufijos que se añaden a columnas con el mismo nombre

2.4 Ejecutar el merge y observar sufijos ⏱ 3 min
Realiza el join y observa qué ocurre con las columnas que tienen el mismo nombre (Name) en ambos DataFrames.
Comportamiento de merge() con columnas homónimas: Pandas añade automáticamente el sufijo _x a la columna del DataFrame izquierdo e _y al del derecho. La columna de cruce (GenreId) no se duplica. Se puede personalizar con suffixes=("_tracks","_genres"). 
# Merge básico — sufijos por defecto _x / _y
df_join = df_tracks.merge(df_genres, on="GenreId", how="left")

# Con sufijos personalizados
df_join = df_tracks.merge(
    df_genres,
    on="GenreId",
    how="left",
    suffixes=("_tracks", "_genres")
)

2.9 Reflexión: SQL vs Pandas ⏱ 2 min
Expón las ventajas e inconvenientes de SQL frente a Pandas y cuándo usar cada uno.
SQL:
  • Más natural para consultas de cruce, agrupación y filtrado sobre datos relacionales.
  • Se ejecuta en el motor de la base de datos (eficiente para grandes volúmenes).
  • Limitado a operaciones declarativas; no puedes usar lógica arbitraria de programación.
Pandas:
  • Una vez los datos están en memoria, las transformaciones encadenadas son muy potentes.
  • Acceso a todo el ecosistema Python (visualización, ML, etc.).
  • Más verboso con los nombres de columnas y las convenciones del merge.
Conclusión: No es una elección excluyente. SQL para extraer y filtrar en la fuente; Pandas para transformar y analizar en memoria.
